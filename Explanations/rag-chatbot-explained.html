What Is RAG?
Retrieval + Generation = grounded answers without hallucination
▼

RAG (Retrieval-Augmented Generation) solves the biggest problem with chatbots: hallucination. A regular LLM might confidently make up an answer. A RAG chatbot first searches for relevant information, then generates an answer only from what it found.

Chunking & Embeddings : Turning Text into Searchable Vectors
14 documents → 254 chunks → 762 vectors in ChromaDB
▼

Why Chunk?

LLMs have limited context windows, and embedding models work best on focused pieces of text. Sending an entire 17,000-character file as one embedding would dilute the signal eg: a question about "Ridge regression" would get a weak match against a document that also discusses 20 other topics. Smaller chunks (800 characters, ~200 words) ensure each chunk is about ONE topic, making retrieval precise.

RecursiveCharacterTextSplitter splits intelligently - it tries paragraph breaks first, then sentence boundaries, then word boundaries. The 100-character overlap ensures no information is lost at chunk boundaries.

Chunk Size Experiment

Chunk SizeTotal ChunksRetrieval Accuracy
400 chars55588% - too small, loses context
600 chars359100%
800 chars ✅254100%

800 characters was chosen as the sweet spot. it achieves 100% accuracy while keeping chunks focused enough for precise retrieval. The 400-char size failed because chunks were too small to contain enough context for the embedding model to understand the topic.

Embedding Model Comparison

ModelDimensionsRetrieval Accuracy
all-MiniLM-L6-v2 ⭐384100%
BAAI/bge-base-en-v1.576888%

The smaller model won! all-MiniLM-L6-v2 was specifically trained for semantic similarity using contrastive learning on 1 billion sentence pairs, making it ideal for retrieval. The larger model captures more information per vector but dilutes the topic-relevance signal across its 768 dimensions. For a focused knowledge base like this, a task-specific lightweight model outperforms a larger general-purpose one.

🏫 School Analogy: Embeddings are like placing every chunk of text on a giant map where similar topics cluster together. "Transfer learning" and "pre-trained MobileNetV2" end up near each other even though they share no words, because the embedding model understands they're semantically related. When you ask a question, it gets placed on the same map, and the nearest chunks are retrieved.

The Custom Prompt

The prompt explicitly constrains the LLM to only answer from retrieved context:

📝 Prompt Template: "You are an ML teaching assistant. Answer the question based ONLY on the following context from ML course materials. If the context doesn't contain enough information to answer, say 'I don't have enough information in the course materials to answer this.'"

This "answer ONLY from context" instruction is what prevents hallucination. Without it, the LLM would fall back to its general training knowledge, which might be wrong or outdated for your specific course content.

Example: Transfer Learning Question

When asked "How was transfer learning used in the X-ray project?", the chatbot retrieved chunks from xray-classification-explained.html and generated an answer citing Model B (Frozen MobileNetV2, 89.9%) and Model C (Fine-Tuned, 87.7%), with the correct explanation of why fine-tuning hurt on a small dataset.

Semantic Similarity : How Embeddings Understand Meaning
Query-query similarity heatmap proves semantic understanding
▼

The semantic similarity heatmap reveals that the embedding model understands conceptual relationships between queries, not just keyword overlap. "What is transfer learning?" and "How does BERT work?" have moderate similarity because both involve pre-trained models, even though they share no words.

📌 Why This Matters: Traditional keyword search would fail here, searching for "transfer learning" wouldn't find a chunk about "pre-trained MobileNetV2 with frozen layers" because they share no keywords. Semantic search via embeddings understands they're the same concept, which is why retrieval accuracy reached 100%.

Summary
▼
🎯 Takeaway #1 RAG Prevents Hallucination. By constraining the LLM to answer ONLY from retrieved context, the chatbot achieved 0% hallucination on out-of-scope questions. Without RAG, the LLM would guess and often get specific details wrong.
🎯 Takeaway #2 Embeddings Understand Meaning, Not Just Keywords. Semantic search found the right chunks for every question even when the query and chunk shared no words. "Transfer learning" matches "pre-trained MobileNetV2" because the embedding model understands they're conceptually related.
🎯 Takeaway #3 Smaller Models Can Win. all-MiniLM-L6-v2 (384 dims) outperformed BAAI/bge-base (768 dims) for retrieval accuracy. The smaller model was specifically trained for semantic similarity, making it a better fit than the larger general-purpose model.
🎯 Takeaway #4 Chunk Size Has a Sweet Spot. 400-char chunks were too small (88% accuracy) — they lacked enough context. 600–1200 chars all achieved 100%. The experiment proved that chunk size matters and should be validated, not guessed.
🎯 Takeaway #5 Negative Testing Is Essential. Proving the chatbot says "I don't know" when it should is just as important as proving it answers correctly. The 6/6 refusal rate on out-of-scope questions demonstrates the system is trustworthy.
🎯 Takeaway #6 RAG Is the #1 GenAI Skill. This project demonstrates the full RAG pipeline — document loading, chunking, embedding, vector storage, retrieval, prompt engineering, and evaluation — which is the most in-demand skill in GenAI job interviews right now.
